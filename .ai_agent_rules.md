# AI Agent Rules

このプロジェクトにおいてAIアシスタント（エージェント）が遵守すべきルールや記憶事項です。

## ドキュメント・作図に関するルール
* アーキテクチャ図やフロー図などの図解を作成する際は、なるべく **Mermaid形式** を使用すること。

## エージェントの運用・対応ルール
* エージェントチームとしてタスクを実施するなど、複数のモデルを利用可能なユースケースでは、可能な限り **Gemini 1.5 Flash等** の安価なモデルを優先して利用し、コストを抑えること。

## 🚨 開発ワークフロー（全プロジェクト共通・絶対遵守）
今後発生するすべての作業（新規機能実装・バグ修正・ドキュメント更新など）において、以下のフローを例外なく守ること。
**※人間のユーザーがこの手順の指示を忘れた場合、作業を開始する前に必ず人間へ手順の確認・問い合わせを行うこと。**

1. **Issue作成**: 作業内容に応じた Issue を GitHub 上に起票する。
2. **作業ブランチ作成**: メインブランチから、Issueに基づく個別の作業ブランチ（例: `feature/issue-XX`, `fix/issue-XX`）を作成する。
3. **作業の実施**: 該当ブランチ内でのみコード修正やレビュー反映を実施する。
4. **Pull Request (PR) の作成**: 作業完了後、メインブランチへ向けて PR を作成する。この際、PRの本文(`body`)には必ず **`Closes #紐づくIssue番号`** を記載し、マージと同時にIssueが自動クローズされる仕組みを徹底すること。
5. **マージは人間が行う**: PR のマージ処理は AI アシスタント側では行わず、**必ず人間のレビューと承認・マージを待つ**こと。

## 🧪 テスト戦略と品質保証ルールの徹底 (全プロジェクト共通)
1. **可能な限りの事前テストの義務**: 実装中は、ユニットテストやローカルエミュレータ（`sam local` 等）を最大限活用し、Macローカルでの検証工程を必ず挟むこと。
2. **Playwrightを活用したE2Eテスト**: ブラウザを介した最終的な統合テスト（E2E）は、手動ではなく必ず **Playwright CLI (`npx playwright test`)** を用いてスクリプト化すること。
3. **再現性の確保**: 作成したテストケースは `docs/` 等にドキュメントとして残し、テストスクリプト自体もリポジトリにコミットし、誰でもいつでも再実行できるようにすること。
