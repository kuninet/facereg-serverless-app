# テスト戦略 (Test Strategy)

本プロジェクト（および今後の関連プロジェクト）におけるテストの基本方針、Macローカル環境での実施範囲、およびPlaywrightを用いたE2Eテストの戦略を定義します。
この方針は**すべての実装ステップにおいて例外なく組み込まれ、再現可能なスクリプトとしてコミットされる**ことが求められます。

## 1. Macローカル上でのテスト (どこまで可能か)

Macローカル環境において、AWS上の実環境にデプロイする前に**「ほぼ100%」の事前検証が可能**です。

### 1.1 バックエンド (AWS SAM / Lambda)
* **ユニットテスト (Vitest / Jest)**:
  AWS SDKをモック（`aws-sdk-client-mock`等）することで、Lambda関数の純粋なビジネスロジック（入力値のバリデーションやTTL計算など）を数ミリ秒でテスト可能です。
* **ローカルAPI統合テスト (`sam local start-api` / `sam local invoke`)**:
  Mac上でAWS SAM CLIとDockerを組み合わせることで、**「本番のLambdaと全く同じAmazon Linuxのコンテナ内」でコードをローカル起動**できます。これによりOS依存の不整合を完全に排除したテストが手元で可能です。

### 1.2 フロントエンド (React / Vite)
* **ユニット・コンポーネントテスト**: `Vitest` + `React Testing Library` を用いて、ブラウザを使わずにコンポーネント単位で超高速にテストします。
* **ローカルサーバー起動**: `npm run dev` でローカルホスト上に展開し、目視および手動でのUIの動作確認が可能です。

---

## 2. E2E (End-to-End) テスト戦略

**「すべての機能は実環境（または統合されたローカル環境）を通して自動テスト可能であること」** を原則とします。手動テストへの依存を減らし、再現性を担保します。

### 2.1 Playwright CLI を用いた自動化の徹底
E2Eテストは、すべて **[Playwright](https://playwright.dev/)** を活用して実装します。

* **再現可能なスクリプト化**:
  テストコードはリポジトリ内の `tests/e2e/` 等の専用ディレクトリに保存し、`npx playwright test` コマンド一発で誰でも再実行な状態を維持します。
* **CIへの組み込みを見据えた設計**:
  Playwrightテストは、将来的にGitHub Actionsなどで自動実行できる形（Headlessブラウザ利用）で作ります。

### 2.2 テストケースの文書化と保存
実装の進行中や手動検証した際に生み出された「テストシナリオ・テストケース」は、単なる脳内の手順で終わらせず、必ず **`docs/テストケース.md` 等として文書化**します。
文書化されたテストシナリオに基づき、Playwrightのテストスクリプト（`.spec.ts` 等）へ落とし込みます。

---

## 3. 今後のワークフローへの組み込み

先ほどの「絶対遵守ワークフロー」に加え、各Issueの実装時（特に実装完了前のPR作成時）には、以下のプロセスが必須となります。

1. **テストコードの実装**: 実装した機能に対応するユニットテストコード（フロント/バックエンド）を作成または更新する。
2. **スクリプトによる検証**: ユニットテストが全てパスすることを確認する。
3. **E2Eスクリプトの作成（機能完了時）**: 画面やAPIが繋がったタイミングでPlaywrightのテストスクリプトを追加する。
