# データベース設計書 (DynamoDB)

本システムはAWSのサーバーレスアーキテクチャを採用しているため、データベースにはAmazon DynamoDBを利用します。要件に基づき、コストを抑えつつ高いパフォーマンスを発揮できるよう**単一テーブル設計 (Single Table Design)** の方針を取ります。

## 1. テーブル概要

| 項目                        | 設定値                                                                         |
| :-------------------------- | :----------------------------------------------------------------------------- |
| **テーブル名**              | `facereg-entries-table` (仮)                                                   |
| **パーティションキー (PK)** | `pk` (String) - エンティティのIDや種別                                         |
| **ソートキー (SK)**         | `sk` (String) - タイムスタンプなどのソート用属性                               |
| **TTL属性**                 | `expires_at` (Number) - UNIXエポックタイム秒。この時間を過ぎると自動削除される |

---

## 2. 項目 (アイテム) 設計

このテーブルには以下のエンティティ（データの種類）を格納します。
受付登録された「顔写真受付エントリー」のデータです。

### 2.1. 受付エントリー (Entry)

| 属性名           | 型     | 説明                                                                                             | 例                               |
| :--------------- | :----- | :----------------------------------------------------------------------------------------------- | :------------------------------- |
| `pk`             | String | (PK) 固定値 `ENTRY` として定義し、GSIを利用せず簡単な一覧取得を可能にする（簡易版）              | `ENTRY`                          |
| `sk`             | String | (SK) ULID または UUID + タイムスタンプ。作成順のソートに使用                                     | `2024-03-12T10:00:00Z#01HGW...`  |
| `id`             | String | エントリーの一意のID                                                                             | `01HGW...`                       |
| `created_at`     | String | 登録日時 (ISO8601形式)                                                                           | `2024-03-12T10:00:00Z`           |
| `company_name`   | String | 会社名                                                                                           | `〇〇株式会社`                   |
| `visitor_name`   | String | お名前                                                                                           | `山田 太郎`                      |
| `purpose`        | String | ご用件（リスト選択値）                                                                           | `商談`                           |
| `purpose_detail` | String | ご用件詳細（任意入力）                                                                           | `新製品のご案内のため`           |
| `photo_url`      | String | S3に保存された顔写真のオブジェクトキー または URL                                                | `photos/2024-03-12/01HGW....jpg` |
| `expires_at`     | Number | **【最重要】 TTL属性。** `created_at` から7日後のUNIXタイムスタンプを指定。AWS側で自動削除される | `1710842400`                     |

※備考: データ量が膨大になる場合は `pk` に日付（例: `ENTRY#2024-03-12`）などを含めてパーティションを分散させる設計が望ましいですが、今回の小規模利用（キオスク運用）の想定では、実装のシンプルさを優先して固定値 `ENTRY` での一覧取得（Query）で十分に対応可能です。

## 3. インデックス設計

基本的にはPK/SKの構成だけで要件の「一覧表示」と「ソート」は満たせます。
UIの要件にある**「検索・絞り込み」**機能については、以下のいずれかで実装します。

1. **(推奨) アプリケーション側でのフィルタリング (コスト最小)**:
   DynamoDBのテーブルに対する `Query` または `Scan` 時に `FilterExpression` を用いて、DynamoDB側で不要なデータを削ってからLambdaに返す。またはLambda/フロントエンドで全件取得（最大でも1週間分のためデータ量は少ない）してから絞り込む。
   *理由: データ保持期間がわずか1週間かつ小規模な来客数であれば、レコード数は非常に少なく、複雑なGSI（グローバルセカンダリインデックス）を作成して課金コストを増やす必要性がないため。*

2. **GSI (必要に応じて)**:
   もし将来的にデータが大規模化した場合にのみ追加を検討します。

---

## 4. TTL (Time To Live) の動作と制約

要件である「1週間での自動削除」を実装するための設定です。
1. アプリケーション（Lambda）からDynamoDBに新規レコードを登録(`PutItem`)する際、現在時刻に7日間（約604,800秒）を足した数値を `expires_at` 属性にセットします。
2. AWS（DynamoDBのバックグラウンドプロセス）がこの `expires_at` を監視し、期限を過ぎたレコードを自動で（課金なしで）削除します。

**【重要制約と対応】**
* DynamoDBのTTL仕様上、`expires_at` を過ぎてから**実際にデータが物理削除されるまで最大48時間のラグ**が発生します。
* そのため、管理画面からAPI（`/admin/entries`）で一覧を取得する際は、単に全件取得するだけでなく、**Lambda側で「現在時刻が `expires_at` を超えているレコードは除外して(フィルタリングして)フロントに返す」** 防御的実装を必ず組み込むものとします。
