# テスト戦略 (Test Strategy)

本プロジェクトにおけるテストの基本方針、Mac ローカル環境での実施範囲、および Playwright CLI を用いた E2E テスト戦略を定義します。
この方針は**すべての Issue 対応で例外なく組み込み、再現可能なスクリプトとしてコミットする**ことを前提とします。

## 0. 基本方針

- 手動テストだけで完了扱いにしない。
- 各 Issue は、実装と同時に対応するテストを追加または更新する。
- 各 Issue は、実行結果とは別に「何を確認するテストか」を記したテスト仕様書を必ず残す。
- 画面導線は Playwright CLI により `npx playwright test` で再実行できる状態を維持する。
- テスト結果はブランチ単位で確認し、PR に実行コマンドと結果を残す。
- テスト結果は git 管理外の `tmp/test-results/` に保存し、保存先パスをユーザーへ明示する。
- 一時生成物やビルド成果物をテスト対象に混ぜない。

## 1. Macローカル上でのテスト (どこまで可能か)

Macローカル環境において、AWS上の実環境にデプロイする前に**「ほぼ100%」の事前検証が可能**です。

### 1.1 バックエンド (AWS SAM / Lambda)
* **ユニットテスト (Vitest / Jest)**:
  AWS SDKをモック（`aws-sdk-client-mock`等）することで、Lambda関数の純粋なビジネスロジック（入力値のバリデーションやTTL計算など）を数ミリ秒でテスト可能です。
* **ローカルAPI統合テスト (`sam local start-api` / `sam local invoke`)**:
  Mac上でAWS SAM CLIとDockerを組み合わせることで、**「本番のLambdaと全く同じAmazon Linuxのコンテナ内」でコードをローカル起動**できます。これによりOS依存の不整合を完全に排除したテストが手元で可能です。

### 1.2 フロントエンド (React / Vite)
* **ユニット・コンポーネントテスト**: `Vitest` + `React Testing Library` を用いて、ブラウザを使わずにコンポーネント単位で超高速にテストします。
* **ローカルサーバー起動**: `npm run dev` でローカルホスト上に展開し、目視および手動でのUIの動作確認が可能です。

## 2. E2E (End-to-End) テスト戦略

**「主要機能は、統合されたローカル環境を通して自動テスト可能であること」** を原則とします。手動テストへの依存を減らし、再現性を担保します。

### 2.1 Playwright CLI を標準とする
E2E テストは **[Playwright](https://playwright.dev/)** の CLI 実行を標準とします。

* **標準実行コマンド**:
  `npx playwright test`
* **再現可能なスクリプト化**:
  テストコードは `frontend/tests/*.spec.ts` に保存し、ローカルでも CI でも同じコマンドで再実行できる状態を維持します。
* **CI 組み込み前提**:
  Headless 実行を前提にし、GitHub Actions 等へ持ち込みやすい構成を維持します。
* **非推奨**:
  単発確認用の ad-hoc なブラウザ操作スクリプトだけに依存しない。恒久テストは `.spec.ts` と Playwright CLI に寄せます。

### 2.2 テストケースの文書化と保存
実装の進行中や手動検証で得られたテストシナリオは、単なる脳内手順で終わらせず、必ず文書またはテストコードに残します。
文書化したシナリオは、優先度の高いものから Playwright の `.spec.ts` や API テストへ落とし込みます。

---

## 3. Issue ごとのテスト運用

各 Issue の実装時、特に PR 作成前には以下を必須とします。

1. **テスト観点の先出し**: 変更前に、何を unit / integration / E2E のどこで守るかを決める。
2. **テストコードの更新**: 実装した機能に対応するテストを追加または更新する。
3. **CLI での検証**: ローカルで再現可能なコマンドを実行し、結果を確認する。
4. **E2E の維持**: UI やユーザー導線に影響する変更では、Playwright CLI の `.spec.ts` を追加または更新する。
5. **テスト仕様書の更新**: 対象 Issue の観点、前提条件、期待結果を `docs/testing/` 配下に残す。
6. **PR への記録**: 実行したコマンドと結果、未自動化の残件を PR に記載する。

## 4. 推奨コマンド

```bash
# backend
cd backend && npm test

# frontend lint
cd frontend && npm run lint

# frontend E2E
cd frontend && npx playwright test
```
