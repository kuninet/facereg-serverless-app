# 設計ドキュメント 多角的な批判的レビュー結果

各専門領域の視点からのレビューを実施し、以下の改善点・リスク事項を指摘しました。これらは設計ドキュメントに反映します。

## 1. 🛡 セキュリティ観点からのレビュー (Security Engineer)
* **指摘事項**: S3署名付きURL (Presigned URL) の発行API (`/entries/initialize`) において、アップロード可能な**ファイルサイズ制限**と**拡張子（MIMEタイプ）の厳格な制限**が要件定義・API仕様から漏れています。
* **リスク**: 悪意のあるユーザーが巨大なファイルや実行可能ファイルを直接S3にアップロードし、ストレージ容量の枯渇（DDoS）やマルウェア拡散の温床となる危険性があります。
* **対応要件**: 
  * AWS Lambdaでの署名付きURL発行時に、`ContentType` と `ContentLengthRange` の条件付きアップロード制限 (POST Policy等を使用) を実装することを明記する。

## 2. 💰 運用・スケーラビリティ観点からのレビュー (SRE / Cloud Architect)
* **指摘事項**: 「1週間での自動削除機能」について、DynamoDBのTTL（Time To Live）を利用する設計となっていますが、DynamoDB TTLは**「期限切れ後、実際に削除されるまでに最大48時間のラグがある」**という仕様があります。
* **リスク**: 「きっちり7日目（168時間後）にデータが物理的に消えなければならない」という厳格な個人情報保護コンプライアンス要件が存在する場合、問題になる可能性があります。
* **対応要件**: 
  * 実削除までに最大48時間の遅延があることを要件定義書の制約事項に明記する。
  * また、API経由での取得時 (`/admin/entries`) には「TTL経過済みのデータはアプリ側でフィルタリングして返さない」処理を入れることをDB設計書に追記する。

## 3. 🎨 UX (ユーザー体験) 観点からのレビュー (UX Designer)
* **指摘事項**: 受付用のキオスクアプリとしてタブレットを不特定多数が使い回す際、ブラウザの「入力履歴（オートコンプリート）」が残っていると、前の人の名前や会社名がサジェストされてしまう情報漏洩リスクがあります。
* **対応要件**:
  * フロントエンドの実装時（UI設計）、入力フォームに対して `autocomplete="off"` を設定し、履歴が絶対に表示されないように徹底することを追記する。
